General

No fer servir any​

No fer servir el tipus any sempre que es pugui evitar. Es pot donar tipus a gairebé tot mitjançant genèrics de TypeScript.

Utilitzar noms descriptius per genèrics​

Pels tipus genèrics s'ha d'utilitzar noms complets que indiquin què és el genèric. Per exemple:

// ❌ Incorrecte: noms de genèrics poc descriptius
function create<A extends Record<string, number>, B extends keyof A>(record: A, key: B): A[B];

// ✅ Correcte: noms de genèrics descriptius
function create<TRecord extends Record<string, number>, TKey extends keyof TRecord>(record: TRecord, key: TKey): TRecord[TKey];

Evitar casts​

Per la mateixa raó que el punt anterior, evitar fer servir casts (p.e. as string). Si s'ha fet un typing correcte, si cal fent servir genèrics, el tipus l'ha de portar la variable.

// ❌ Incorrecte: fer servir cast
const name = getValue() as string;

// ✅ Correcte: el tipus prové de la funció genèrica
const name = getValue<string>();

Partial<Record<>> en cas de keys incomplets​

Quan es fa servir Record<number, IObject>, tenir en compte que TypeScript assumeix que tots els keys de number estaran definits.

Per seguretat en el tipatge, s'ha de fer servir sempre Partial<Record<number, IObject>> quan els keys puguin no ser complets (casos de strings, numbers).

// ❌ Incorrecte: assumeix que tots els keys existeixen
const map: Record<number, IObject> = {};
const value = map[42]; // TypeScript creu que sempre és IObject

// ✅ Correcte: indica que els keys poden no existir
const map: Partial<Record<number, IObject>> = {};
const value = map[42]; // TypeScript sap que pot ser IObject | undefined

Exhaustiveness checks​

Aplicar sempre que es pugui exhaustiveness checks, per evitar que canvis en els possibles valors d'una variable o enum provoquin errors de runtime quan es fan comprovacions condicionals sobre tots els possibles valors.

Això es fa fent servir el tipus never.

enum EStatus {
    Active = "Active",
    Inactive = "Inactive",
}

function handleStatus(status: EStatus): string {
    switch (status) {
        case EStatus.Active:
            return "Actiu";
        case EStatus.Inactive:
            return "Inactiu";
        default:
            // Exhaustive check
            const _exhaustiveCheck: never = status;
            return _exhaustiveCheck;
    }
}

Preferir Record per mapejar valors​

Per casos en què es vol assignar un valor a cada opció d'un conjunt finit de keys, és preferible fer servir un Record en lloc d'una funció amb switch. Un Record és més concís, més fàcil de mantenir i, combinat amb el tipatge de TypeScript, garanteix que totes les claus estiguin cobertes en temps de compilació.

En cas d'utilitzar un switch, sempre utilitzar exhaustiveness checks.

// ❌ Incorrecte: funció amb switch verbose i propensa a errors
const getTaskTypeColor = (taskType: ISuperuserTasksSchema["type"]): string => {
    let color;
    switch (taskType) {
        case "train":
            color = IdsUtils.Colors.Other.green1;
            break;
        case "package":
            color = IdsUtils.Colors.Other.orange1;
            break;
        case "inference":
            color = IdsUtils.Colors.Other.blue1;
            break;
        case "test":
            color = IdsUtils.Colors.Other.red1;
            break;
        default:
            // ❌ Incorrecte: falta exhaustiveness check
            color = IdsUtils.Colors.Gray.light5;
    }

    return color;
};

// ✅ Correcte: Record que mapeja directament cada clau al seu valor
const TASK_TYPE_COLOR: Record<ISuperuserTasksSchema["type"], string> = {
    train: IdsUtils.Colors.Other.green1,
    package: IdsUtils.Colors.Other.orange1,
    inference: IdsUtils.Colors.Other.blue1,
    test: IdsUtils.Colors.Other.red1,
};
