Classes VisionElement

Propòsit dels tipus VisionElement​

Què permeten els tipus VisionElement?

  • Tracking amb identificador únic (ID).
  • Possibilitat de fer aniuament d'objectes (parent/children).
  • Serialització (guardar/obrir objectes des de disc).
  • Propietats de visualització (colors, gruix de línia, etc).
  • Tipus base únic VisionElement per manejar els objectes (polimorfisme). El problema és que no podrem fer std::vector<VisionElement>.

Què pot ser un Vision Element​

Un tipus ha de ser un Vision Element si es compleix alguna d'aquestes dues condicions:

  • Té sentit mostrar-lo sobre una imatge.
  • O representa un resultat d'una operació de visió que voldrem guardar o analitzar, i no hi ha algun dels Vision Elements actuals que el pugui representar per si sol.

Sempre s'ha d'intentar que el VisionElement representi la mínima informació possible, la mínima unitat indivisible. Per exemple, pel Pattern Matching segurament serà millor crear el VisionElement Keypoint2d, i no crear un VisionElement que contingui una llista de keypoints. La mínima unitat indivisible és el Keypoint2d.

Tipus que no han de ser Vision Elements:

  • Primitives.
  • Matrius.
  • Objectes de configuració.

Justificació de tipus que són Vision Elements però estan a l'extrem:

  • Tensor: el fem Vision Element perquè és el resultat d'una operació de visió (xarxa neuronal) i necessitem la funcionalitat de guardar, aniuar i accés des de classe base VisionElement.
  • CameraCalibration (antic DistortionData): és un resultat de visió.

Value i props​

Un VisionElement està format per basics + value + props.

  • Basics: són les propietats comunes que comparteixen tots els VisionElements (id, type, parent_id, name, description).
  • Value: serveix com a emmagatzematge dels valors intrínsecs de l'element. Aquests són els valors que serveixen per fer operacions amb l'element, siguin geomètriques o de qualsevol altre tipus. Per exemple, d'un punt els valors intrínsecs són {x, y}.
  • Props: serveixen per propietats que no són intrínseques de l'element, sinó que són complements, per exemple per pintar l'element en una pantalla.

Els VisionElements són smart pointers de la classe IVisionElement, i són els elements que exposem a la interfície pública. Exposem shared pointers de IVisionElement a la interfície pública degut a la vinculació que tenim tant des de OneVision com des dels scripts de runtime amb el llenguatge TypeScript, en el qual els objectes són més o menys equivalents als shared pointers. (Realment els shared pointers són millors que el garbage collector de JavaScript).

Mètodes classe base​

Quan parlem de mètodes, són funcions que pertanyen a una classe (funcions membre). Les funcions lliures (o stand-alone) no es poden dir mètode.

El mètode clone és un mètode abstracte comú per tots els Vision Elements, perquè és patró habitual en implementacions de classes amb polimorfisme. L'objectiu del clone és poder generar una còpia d'aquella instància, sigui cridant el clone des de la classe base com des de la classe derivada.

Mètodes classes derivades​

Per a tots els VisionElements, és preferible evitar mètodes get/set i fer públiques les variables del value directament. Això és perquè a nivell d'usuari és més pràctic fer point.x = 3 que point.setX(3). I també perquè estalvia codi.

Actualment les classes Shape i Path funcionen amb get/set per una limitació del Node-addon API, que no permet implementar els getters i setters de forma niuada per elements d'una llista. Si no hi hagués aquesta limitació, també exposaríem directament els vectors de punts tant pel Shape com pel Path.

note

Quins mètodes poden formar part de les classes derivades de VisionElement? No hi volem funcions que requereixin molt de processament, només consultes sobre la informació de la instància.

Quina classe utilitzar a la implementació​

Les funcions d'implementació han d'utilitzar els elements més bàsics possible, que normalment són els value. Per exemple, per moure un punt, l'element més bàsic és el value (no utilitzarem directament x i y). En canvi, per convertir un element a JSON l'element més bàsic és directament el VisionElement.

Diferència Shape2d i Path2d​

Els paths defineixen una trajectòria. Els shapes defineixen una àrea.

Els shapes, definint una àrea, poden tenir forats.

Serialització​

Flat​

L'objectiu dels elements flat és representar exactament la mateixa informació que els VisionElement natius, però en forma de propietats accessibles/públiques. O sigui que exposen el valor de forma pública, no amagat dintre d'una classe.

Amb els flat no es podran fer operacions, perquè no són una instància d'una classe, però sí que ens permeten enviar dades.

Només es permeten els tipus: object, list, number, boolean, string.

note

No tenim clar si volem que els flat es mantinguin dintre de la llibreria, si són prescindibles o superflus.

JSON​

La conversió a JSON s'ha de fer de manera que si des de TypeScript fem:

flat = ve_to_flat(ve);
str1 = JSON.stringify(flat);
str2 = ve_to_json(ve);
// str1 i str2 han de ser el mateix
