Naming

Prefixos en tipus de variables​

TipusPrefixExemple
EnumsEEVisionElement
Classes, structsNo utilitzar prefix I o S (antigament s'havia fet)FitStats

Distingir objectes VisionElement i altres​

Cal distingir entre objectes (classes o structs) que són VisionElement i els que no.

Per fer-ho, fem servir la següent política:

  • Les classes VisionElement poden tenir el nom que sigui.
  • Els objectes de resultats que retornen les funcions (objectes per contenir múltiples resultats) han d'acabar amb Result, per exemple: FindEdgeResult, FitEllipseResult.
  • Els objectes de parametrització de funcions (quan una funció té molts paràmetres i els volem compactar en un sol objecte) han d'acabar amb Config: FitEllipseConfig.
  • Les classes utilitat (per exemple la classe per rectificar imatges) han d'acabar amb Utility: RectificationMapUtility.

Ordre de paraules en les funcions​

No començar el nom d'una funció amb el nom del VisionElement al qual aplica, fer servir sempre primer l'acció o propietat. Això serveix perquè els assistents de programació que et mostren la llista de funcions disponibles, mostrin les funcions relacionades agrupades.

Per exemple, no fer point_scale, rect_scale... sinó scale_point, scale_rect, scale_circle.

Ordre de VE en funcions​

Quan una funció involucra arguments VisionElement heterogenis i no hi ha un argument més important que d'altres (per exemple en funcions simètriques, com distance_line_point), ordenar els dos elements alfabèticament.

Per exemple: distance_point_shape, intersect_line_path.

En el cas de que hi hagi una direccionalitat, o un element fort, posar primer l'element fort. Per exemple, sempre posar primer el src i després el dst.

Fer servir paraules completes​

No fer servir abreviacions, fer servir les paraules completes. Per exemple, calib_camera() no, seria calibrate_camera().

Fer servir la forma plural del nom de l'element​

Quan estem operant sobre dos elements del mateix tipus de VisionElement, posem l'acció (o verb) i el nom dels elements que estem operant. Per exemple: add_shapes (no fer add_shape).

Modificadors en variants de funcions​

Quan es creïn variants de funcions a partir de la funció canònica, afegir els modificadors al final del nom.

Per exemple, la funció distance_path_path_points és una variant de la funció distance_path_path que pren una llista de punts enlloc d'un VE path. El modificador _points ha d'anar al final de la funció.

Concretament fer servir el modificador _list, quan una funció es caracteritzi per tenir com a argument una llista d'algun element. No fer servir _array o _vector.

Noms preferits per accions​

Sempre preferim verbs sobre noms per les accions. Per exemple, intersection_shape seria incorrecte, és millor intersect_shape. Si no és un verb, ha de ser un símil d'un verb, per exemple: xor_shapes(). Xor es considera un verb, perquè podem dir "xor something".

Noms preferits: get, create, set, reset, convert, move, rotate, sum, threshold, find.

Noms preferits per accions sobre llistes​

NomDescripció
removeEliminar un element mitjançant índex o valor
addAfegir element al final de la llista (enlloc de push o append)
insertAfegir element en una posició concreta de la llista
popEliminar últim element de la llista
findBuscar element en la llista
numRetornar el número d'elements de la llista. No fer servir length, perquè es podria confondre amb la llargada geomètrica

Noms preferits per propietats​

NomDescripció
isSi es compleix una condició
hasSi té un element
lengthLlargada geomètrica (normalment en píxels), no fer servir per elements d'una llista
angleAngle
areaÀrea
center_massCentre de massa
maxMàxim
minMínim
originOrigen
centerCentre

En el cas de num, que es fa servir per exemple en casos com num_pixels, fer servir només si sense aquest prefix el propòsit de la funció no queda clar. Per exemple, fer servir en el cas de num_channels: sense num, channels es podria pensar que retorna alguna cosa com RGB. En canvi no fer servir en el cas de bytes_per_pixel. num_bytes_per_pixel seria redundant.

Funcions que retornen booleans​

Les funcions que retornen booleans com a resultat principal, preferiblement han de tenir el prefix is o has.

Per exemple, fer servir is_valid_contour, i no valid_contour.

Eliminar el postfix _2d​

Elimineu la part 2d del nom de les funcions. Exemple: area_rect, enlloc de area_rect_2d. Aquest postfix està bé pel nom de la classe VisionElement, per distingir-la de la 3d, però és redundant en el nom de les funcions.

Si hi ha col·lisió en algun cas amb funcions de 3d, potser afegirem el postfix 3d.

Noms de paràmetres habituals​

NomSignificat
srcSource
dstDestination
veVisionElement
listLlistes, més genèric que array o vector, que són termes més específics de llenguatges

Funcions d'implementació​

Les funcions de la implementació tenen el mateix nom que les funcions públiques afegint _ al final.

Nota sobre la diferència entre paràmetres i arguments d'una funció​

Quan parlem de paràmetres ens estem referint a les variables de la definició d'una funció.

void print(int x, string message) { // x i message són PARÀMETRES
    cout << message << ": " << x << endl;
}

Quan parlem d'arguments ens referim als valors d'aquella variable en una crida de la funció.

int main() {
    print(42, "The answer"); // 42 i "The answer" són ARGUMENTS
}
